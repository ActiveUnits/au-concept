Introduction

This web page is a concept description of open source innovation project named Active Units (formly known as ABN).

Abbreviations

  • API = application program interface
  • LIB = source code library of classes structured within namespaces.
  • Unit = software application instance (service, web app, mobile app & etc.)

Description

SOA,SAAS

Service-oriented architecture (SOA) is a flexible set of design principles used during the phases of systems development and integration in computing. A system based on a SOA will package functionality as a suite of interoperable services that can be used within multiple separate systems from several business domains. [1]

Software as a service (SaaS, typically pronounced [s√¶s]), sometimes referred to as "software on demand," is software that is deployed over the Internet and/or is deployed to run behind a firewall on a local area network or personal computer. [2]

In regards to SOA & SAAS Active Units concept represent a flexible set of design principles also including ready to use web services & pre-build facilities used to develop Internet based software systems.

Web service

A Web service is a method of communication between two electronic devices. The W3C defines a "Web service" as "a software system designed to support interoperable machine-to-machine interaction over a network. [3]

In regards to web services definition AU concept maps those as "units" and thereafter includes also user interface software applications clients under the same term.

Active Units Network

By definition Active Units Network is an software system of units designed to solve specific problem.

Units

Units are building parts of the Active Units Network working as a suite of interoperable services & end-user software applications. Units as software applications are single-purpose solutions build on top of existing technologies/languages including but not limited to:

  • php
  • ruby
  • java
  • python
  • javascript

Units as web services within SOA-like Active Units Network use an Internet communication protocols best suitable to the unit's own purpose. There are no unified or standartized units' communication protocol and structure due the very different design & purpose of them. The following however proved to satisfy most of the requirements in real world units implementation:

  • http
  • xmpp
  • tcp
  • rabbitMQ
  • dnode

The general purpose of having Active Units based software solutions build on "units" is to be able to re-use particular units for solving specific application problems & requirements.
